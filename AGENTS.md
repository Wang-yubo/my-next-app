# 用户自定义交互规则

## 对话确认流程
每次对话都需要遵循以下流程:

1. **需求复述**: 在开始任何工作前,必须先向我复述一遍您理解的需求,确保我们达成共识
2. **疑问澄清**: 如果对需求有任何疑问、不确定之处,或者有更好的实现建议,必须主动向我提问
3. **等待确认**: 待我明确确认后,才能开始修改代码或执行操作
4. **分步执行**: 对于复杂任务,应该分步骤进行,每完成一步都要向我汇报进度并确认下一步计划

## 重要原则
- 不要假设我的意图,有疑问就问
- 提供多个方案时,说明各自的优缺点,让我做选择
- 修改代码前,先说明将要做什么改动以及为什么
- 如果发现潜在问题或风险,及时提醒我

## 代码重构与优化原则
在进行代码重构、性能优化或架构设计时,遵循以下三个核心原则:

### 1. 不要过早优化
- 在确实遇到性能问题或复用需求之前,保持简单
- 小数据量场景下(如分页每页10条),微优化带来的收益可以忽略不计
- 优先保证功能正确性和代码清晰度,而非极致的性能
- 只有在实际测量到性能瓶颈时,才进行针对性优化

### 2. 渐进式重构
- 随着需求变化逐步调整代码结构,而不是一开始就设计过度复杂的架构
- 采用演进式设计:从简单实现开始,根据实际需求逐步抽象和提取
- 例如 columns 的组织策略:
  - 当前阶段:保持组件内定义,保证可读性
  - 未来如需复用:使用工厂函数传入依赖
  - 复杂动态逻辑:保留在组件内,配合 useMemo 优化
  - 多页面共享:提取到独立文件

### 3. 可读性优先
- 代码首先是给人看的,其次才是给机器执行的
- 局部常量定义在组件顶部,方便快速查看和维护
- 避免过度抽象导致的理解成本增加
- 在性能提升不明显时,优先选择更易读的实现方式
